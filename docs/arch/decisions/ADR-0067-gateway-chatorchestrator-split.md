# ADR-0067: Gateway God Class 拆分 (ChatOrchestrator)

## 状态
Proposed (本阶段仅进行设计论证，暂不进入实现)

## 背景 (Context)
目前系统的入口控制层 `internal/gateway/server/chat/sse.go` 承载了过多的业务职责，已经逐渐演变成了一个庞大的 God Class。它不仅负责底层的网络与协议处理（例如 HTTP 请求验证、SSE 连接维持、分块传输），还过度耦合了复杂的业务逻辑，包括：
1. **输入预处理**：例如 `@file` 引用的上下文展开（ContextRefExpander）。
2. **上下文管理与压缩**：会话历史的读取，以及调用 `Compressor` 进行长上下文的滚动压缩。
3. **消息持久化**：通过 `SaveMessage` 等接口频繁对 DB/Repository 进行操作。
4. **Slash 命令解析**：内置 `SlashRouter`，直接解析并拦截部分系统指令（如 `/compact` 等）。

这违反了网关层的薄网关原则，并触犯了架构设计规范（GD-13-008 职责过载 / GD-9-001 边界污染）。在系统逐步演进（尤其是新增复杂的多代理协作和工作流处理）的过程中，这部分热点代码的回归面大且容易引入 bug。

## 决策 (Decision)
我们决定对现有的 Gateway God Class 进行拆分，引入一个独立的核心编排层组件：`ChatOrchestrator`（或 `AgentSessionFacade`）。
该组件的目标是：
- 接管所有的输入预处理（文件展开、斜杠指令分发）。
- 接管上下文的拼装、压缩策略的调度以及历史消息的持久化保存。
- 让 Gateway 层回归“薄网关”的本职工作，仅负责：网络协议（HTTP/WebSocket）、身份鉴权、限流、路由分发以及 SSE 格式的编解码。

### 落地拆解 (Phases)
由于 `sse.go` 是系统的高频次调用的核心热路径（hot path），如果一次性进行 big-bang 级别的重构，存在极大的系统不可用风险。因此本拆分计划分为多步实施：

- **第一步：消除 Gateway 层的存储越界（A-4 SQL 下沉）**
  - **状态**：已完成。
  - **说明**：通过将直接暴露在 Gateway 层的原始 SQL 下沉到 DAO (Repository) 层，切断了 Gateway 与底层存储引擎的直接耦合。
- **第二步：统一入站分发渠道（A-2 渠道聚合）**
  - **状态**：已完成。
  - **说明**：将不同平台（Slack, Discord 等）和前端页面的入口统筹到统一的 Dispatch 机制中。
- **第三步：抽取 ChatOrchestrator 核心逻辑**
  - **状态**：**待排期（本轮不实施）**。
  - **说明**：此步骤将重写 `sse.go` 的流程引擎，将所有的业务领域逻辑（压缩、持久化、斜杠路由）剥离至新的 `ChatOrchestrator` 接口实现中。网关层只保留对 `ChatOrchestrator` 的方法调用。

## 结果 (Consequences)
**正面影响：**
- 极大地降低了网关层的圈复杂度和代码行数，增强了可读性和可测试性。
- 业务处理逻辑（`ChatOrchestrator`）可以摆脱协议层依赖，便于开展脱离网络的单元测试或集成测试。
- 符合整体系统微核化演进路线。

**负面影响及应对：**
- 需要在重构期间保障主链路稳定。由于将核心网关层的重构推迟至第三阶段执行，目前能够确保短期的业务平稳过渡，代价是短时间内 `sse.go` 依然会保留现有结构。
