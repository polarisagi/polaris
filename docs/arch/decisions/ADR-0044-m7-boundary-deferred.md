# ADR-0044: M7 模块边界拆分（GD-13-002）暂缓，记录待办

- **状态**: Deferred（评估完成，本轮不实施）
- **日期**: 2026-07-11
- **决策者**: MrLaoLiAI
- **相关模块**: M07 / `internal/tool` / `internal/sandbox` / `internal/extension`

## 上下文

Gemini 设计评审（`local_playground/reports/gemini-review-design.md` GD-13-002）提出：`internal/extension`（L2/L3 扩展注册与运行时）当前直接引用/定义部分本应归属 `internal/tool`、`internal/sandbox`（L1）的接口类型，存在跨层"物理边界污染"，建议拆分为独立的包边界。

## 决策

本轮（任务书 1-8 缺陷修复 + 设计裁决批次）**不实施**该拆分，理由：

1. 纯目录/包结构重构，本身不修复任何已知缺陷、不产出新功能收益。
2. 与同批次任务书 2（`02-接口架构统一.md`，接口下沉到 `internal/protocol/` 的重构）在同一批文件上操作，两者若并行推进会显著提高改动冲突与回归风险。
3. CLAUDE.md 明确要求"禁止顺手重构未损坏内容"——当前尚未有真实的循环依赖编译失败、或运行时故障可以证明这层"边界污染"造成过实际损害，收益停留在理论层面。
4. 代价（大范围 import 路径改动，涉及文件数量未知，需要专门评估）是确定的，收益是不确定的，不满足本轮"缺陷修复优先"的批次目标。

## 后果

- 正向：本轮改动面保持收敛，不引入额外的大范围重构风险。
- 负向：`internal/extension` 对 `internal/tool`/`internal/sandbox` 类型定义的跨层引用现状继续保留，未来若确实演变为循环依赖故障，需要单独排期评估拆分方案。

## 重新评估触发条件

未来若有人重提本议题，需先给出以下之一作为量化证据，而不是理论推演：

- 一次因当前包边界导致的真实循环依赖编译失败（`go build` 报错记录）。
- 一次因边界污染导致的运行时故障（可复现的 bug 报告 + 根因分析）。
- 明确的新功能需求，该功能在当前包边界下无法实现或实现成本显著高于拆分后。

满足以上任一条件后，方可重新排期评估拆分方案，且应与其他大范围重构（如接口下沉）错开批次，避免同批次冲突。
